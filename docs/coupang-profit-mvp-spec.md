# 쿠팡 순이익 분석 도구 — MVP 라이트 스펙

> 솔로 개발 / "만들며 검증" 모드. 이 문서의 목적은 **안 만들 것을 정하는 것**.
> 의사결정이 헷갈릴 때마다 여기로 돌아온다.

---

## 1. 한 줄 정의

쿠팡 셀러가 **상품별 진짜 순이익(마진율)**을 자동으로 보는 대시보드.
매출이 아니라 "수수료·원가·비용 다 빼고 실제로 얼마 남았나"를 보여준다.

---

## 2. 스코프 (가장 중요)

### ✅ MVP에 넣는다
- 쿠팡 **단일 채널만** 연동 (API 키 입력 방식)
- 주문 데이터 + 정산(지급) 데이터 자동 수집
- 상품별 **매입원가(COGS)** 셀러 직접 입력
- 상품별 / 기간별 **순이익·마진율** 대시보드
- **적자 상품 자동 하이라이트** (← 핵심 세일즈 포인트)
- 월 구독 결제 (베이직 1종부터)

### ❌ MVP에 안 넣는다 (유료 고객 생긴 뒤로 미룬다)
- 네이버·11번가 등 **다채널 연동**
- 광고비(ROAS) 자동 연동 → MVP는 셀러가 총액 수동 입력
- 자동 발주 / 재고 관리 / 상품 등록
- 리뷰·CS 관련 기능 (네이버는 API로 리뷰 미지원이기도 함)
- 모바일 앱 (웹 반응형으로 충분)
- 팀/다중 사용자, 권한 관리

> 원칙: "있으면 좋은" 기능은 전부 ❌. **적자 상품을 보여주는 것** 하나만 완벽하게.

---

## 3. 핵심 화면 흐름

1. **회원가입 / 로그인** (이메일)
2. **쿠팡 API 키 연동** — 업체코드 / Access Key / Secret Key 입력
3. **상품 원가 입력** — 수집된 상품 목록에 매입원가(COGS)·기타 비용 입력
4. **대시보드** — 기간 선택 → 총매출 / 총순이익 / 평균 마진율 + 상품별 표(적자 빨강 표시)
5. **(설정)** 구독 관리

전체 화면 5개 이내. 이 이상 늘어나면 스코프를 의심한다.

---

## 4. 순이익 계산 로직 (제품의 심장)

쿠팡 **정산 데이터를 신뢰**한다. 정산 지급액은 이미 판매수수료가 차감된 실수령액이라,
수수료를 직접 재계산하지 않는다 (정확도·유지보수 측면에서 유리).

```
상품별 순이익
  = Σ(정산 지급액)                    ← 쿠팡 정산 API (수수료 차감 후 실수령)
  − Σ(판매수량 × 매입원가)            ← 셀러 입력 (COGS)
  − 배분된 기타 비용                   ← 셀러 입력 (광고비·배송비 등, 기간 총액을 매출 비율로 배분)

마진율 = 순이익 / 총매출 × 100
```

- 반품/취소: 정산 데이터에 음수로 반영되므로 자연스럽게 차감됨
- 부가세: MVP에서는 "공급가 기준" 토글 하나로 단순 처리 or 일단 제외 (검증 후 결정)

## 5. 데이터 모델 스케치 (→ DB 스키마 초안)

| 엔티티 | 핵심 필드 | 비고 |
|---|---|---|
| **User** | id, email, password_hash, 구독상태, 가입일 | |
| **MarketAccount** | id, user_id, channel(=coupang), vendor_id, access_key🔒, secret_key🔒 | **키는 반드시 암호화 저장** |
| **Product** | id, market_account_id, vendor_item_id, 상품명, cogs(매입원가), 활성여부 | cogs는 셀러 입력 |
| **Order** | id, market_account_id, 주문번호, product_id, 수량, 판매가, 주문일, 상태 | 쿠팡 주문 API |
| **Settlement** | id, market_account_id, product_id, 지급액, 수수료, 정산일 | 쿠팡 정산 API |
| **Cost** | id, user_id, 유형(광고/배송/기타), 금액, 기간시작, 기간종료 | 셀러 입력, 매출 비율로 배분 |

🔒 = API 키/시크릿은 평문 저장 금지. 애플리케이션 레벨 암호화 후 저장.

## 6. 기술 스택 (확정)

- **백엔드**: Spring Boot — 스케줄러로 주문/정산 주기 동기화
- **DB**: PostgreSQL
- **프론트**: React / Next
- **배포**: 초기엔 단일 서버(예: 가벼운 클라우드 인스턴스)로 충분

## 7. "만들며 검증" — 성공/실패 판단 지표

코드 양이 아니라 **이 신호**를 본다:

- 🟢 베타 셀러가 대시보드 보고 "어 이 상품 적자였네?" 하고 놀란다 → 가치 증명
- 🟢 한 번 쓰고 다음 달에 또 들어와서 본다 (재방문) → 진짜 needs
- 🟢 "월 ○○원이면 쓸게요" 가 말이 아니라 **결제**로 나온다
- 🔴 가입은 하는데 원가 입력하다 이탈 → 온보딩이 무겁다는 신호, 단순화
- 🔴 한 번 보고 다시 안 옴 → 통증이 약하거나 인사이트가 약함

**1차 목표**: 유료 셀러 10명. 여기까지 가면 다채널·광고비 연동으로 확장 시작.
