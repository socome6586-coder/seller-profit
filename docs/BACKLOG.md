## B1. 네이버 스마트스토어 채널 추가

- **기록일**: 2026-07-04
- **아이디어**: 쿠팡뿐 아니라 네이버 스마트스토어도 연동해 "진짜 순이익"을 다채널로 보여줌.
- **판단**: **보류하되 우선순위 높음(배달 아이디어보다 먼저 검토).** 이유가 다르다 —
  이건 새 고객군이 아니라 **기존 쿠팡 셀러가 이미 갖고 있는 다른 채널**이라 제품 완성도
  문제에 가깝다.
- **아키텍처 근거**: `domain/type/Channel.java` 가 이미 `MVP는 쿠팡만, 이후 NAVER 등 확장`
  으로 주석돼 있어 처음부터 멀티채널을 염두에 둔 설계다. `coupang/` 패키지 옆에 같은
  패턴(`naver/`)으로 인증·수집·스케줄러를 추가하면 되므로 **재설계 비용은 없다.**
- **보류 이유**:
  1. 실 검증에는 결국 **본인 스마트스토어 또는 베타 셀러의 실 통합매니저 계정**이 필요 —
     쿠팡과 같은 "사업자등록 후 실 키 검증" 게이팅과 동일 처지(`CLAUDE.md` §B 참고).
  2. 작업량이 "필드 추가" 수준이 아니라 **쿠팡 연동 전체(인증·DTO·수집 스케줄러)와
     맞먹는 신규 통합**이다.
  3. 지금은 쿠팡 단일 채널조차 유료 검증 전이라, 두 번째 채널보다 **배포 + 베타 검증**이
     선행돼야 한다.
- **저비용 사전 검증 방법(즉시 가능)**: 회원가입/랜딩에 "스마트스토어도 운영하시나요?"
  체크박스나 짧은 설문을 추가해, 코드 없이 베타테스터의 실수요를 먼저 잰다.
- **재검토 시점**: 쿠팡 단일 채널로 베타 셀러 지불의사 검증(최소 1건 실 결제 또는 명확한
  "낼게요" 신호) 확보 직후. 위 사전 설문에서 스마트스토어 병행 비율이 높게 나오면 그 즉시
  앞당겨 검토.

## B2. 배달 자영업자(배민/쿠팡이츠) 광고 ROI 서비스

- **기록일**: 2026-07-04
- **아이디어**: seller-profit 과 같은 wedge("ROAS 높은데 팔수록 손해")를 배달앱 사장님
  대상으로. 배민 "우리가게클릭" 광고로 매출은 늘어도 실소득이 줄어드는 사례가 언론
  보도로 확인돼, 페인포인트 자체는 실재.
- **판단**: **보류, 우선순위 낮음(완전히 새로운 제품/고객군).**
- **보류 이유**:
  1. 데이터 접근성 불확실 — 배민/쿠팡이츠에 쿠팡 Open API 급 공식 API 가 있는지 미확인.
     사장님 웹 화면(배민셀프서비스)만 확인됨 → 자동화라는 핵심 가치가 줄어들 수 있음.
  2. 도메인 불일치 — 조민석 님은 이커머스 셀러 경험은 있으나 요식업 매장 운영 경험 없음.
     seller-profit 을 고른 근거(불공정한 우위)가 여기선 없음.
  3. seller-profit 자체가 유료 고객 0명인 상태에서 **완전히 다른 고객군의 두 번째 제품**을
     시작하는 건 솔로 개발자의 집중력을 분산시킨다(B1 과 다르게 기존 고객 확장이 아님).
- **재검토 시점**: seller-profit 이 유료 전환(베타테스터 중 실 결제 최소 1건) 이후,
  "같은 패턴을 다른 자영업 버티컬로 확장할 가치가 있는가"를 재판단할 때 후보로.
