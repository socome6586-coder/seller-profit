# 셀러프로핏 (seller-profit) — 포트폴리오

> 쿠팡 셀러의 **상품별 진짜 순이익(마진율)** 을 자동 계산해, **적자 상품을 맨 위로 끌어올려** 보여주는 SaaS MVP.
> 매출이 아니라 "수수료·원가·기타비용 다 빼고 실제로 얼마 남았나"가 핵심 가치다.

- **기간**: 1인 개발 (백엔드·프론트·인프라 전 영역)
- **저장소**: https://github.com/socome6586-coder/seller-profit
- **상태**: 로컬 E2E 동작하는 MVP. 시드 데이터로 쿠팡 키 없이 전 기능 시연 가능, 실 쿠팡 API 연동 입구까지 완성(라이브 키 검증 단계).

---

## 1. 문제 정의 — 왜 만들었나

쿠팡 셀러는 매출은 쉽게 보지만 **상품 하나가 실제로 돈을 벌고 있는지**는 잘 모른다. 이유:

- 매출(판매가)과 **실수령액(정산)** 이 다르다 — 판매수수료·배송비·쿠폰이 빠진다.
- **반품/취소** 가 일어나면 매출은 줄고 원가(이미 산 재고)는 그대로 남는다.
- **광고비·기타비용** 은 상품별로 따로 떨어지지 않아 "이 광고가 이 상품을 적자로 만들었는지" 보이지 않는다.

그래서 "매출 1등 상품이 사실은 **적자**" 인 상황이 흔하다. 이 도구는 그 적자 상품을 **자동으로 1순위 노출**한다.

```
진짜 순이익 = 정산 실수령액 − (판매수량−반품수량)×개당원가 − 매출비율로 배분된 기타비용
```

## 2. 핵심 화면 / 기능

| 화면 | 핵심 |
|------|------|
| **로그인/가입** | 이메일+BCrypt, 가입 즉시 자동 로그인. 데모 계정 힌트 노출. |
| **순이익 대시보드** | 총매출(정산 실수령)·총순이익·평균 마진율·배분 기타비용 카드 + 상품별 순이익 표. **적자 상품은 빨간 배경 + `적자` 뱃지로 맨 위 고정.** |
| **반품 사유 분석** | 반품 사유별 수량·건수·비중(막대) 집계. |
| **입력/관리** | 셀러가 직접 상품 원가(COGS)·기타비용(광고/배송/기타)을 입력 → 저장 즉시 순이익 재계산. |
| **계정 연동** | 쿠팡 vendorId/Access/Secret 키 등록(AES-GCM 암호화 저장). 플랜 한도(FREE=1계정) 강제. "지금 동기화" 버튼. |
| **요금제** | FREE/PRO 카탈로그 + 구독 상태. 토스페이먼츠 빌링 스캐폴딩. |

## 3. 아키텍처

```
[React SPA (Vite)]  ──build──▶  src/main/resources/static/
       │  같은 오리진(:8088) → JSESSIONID 세션쿠키, CORS 불필요
       ▼
[Spring Boot 3.4 / Java 21]
  ├─ auth        세션 인증(BCrypt, HttpSession), CurrentUser
  ├─ account     연동/소유권 가드(AccountAccess), 수동 동기화
  ├─ coupang     쿠팡 Open API 연동(HMAC 서명, 주문/정산/반품 수집)
  ├─ profit      순이익 계산(기타비용 매출비율 배분) + 대시보드 API
  ├─ manage      셀러 직접입력(원가/기타비용)
  ├─ subscription 플랜 카탈로그/구독 상태(PlanType 한도)
  ├─ billing     토스페이먼츠 정기결제 스캐폴딩
  └─ crypto      AES-256-GCM 키 암복호화 (@Convert 투명 처리)
       ▼
[PostgreSQL]  Flyway V1~V3, 멱등 UNIQUE, ON DELETE CASCADE
       ▲
[Scheduler]  주문 30분 / 정산·반품 1시간 / 빌링 매일 03:10 KST
```

**데이터 흐름**: 쿠팡 API(HTTP, 트랜잭션 밖) → 페이지 단위 멱등 영속화 → 네이티브 집계쿼리(`findProfitByPeriod`)가 주문·정산·반품·원가·기타비용을 한 번에 조인해 상품별 순이익 산출 → 대시보드.

## 5. 설계에서 신경 쓴 지점 (엔지니어링 포인트)

### 5.1 보안 — 계정 열거(enumeration) 차단
소유권 실패와 리소스 없음을 **동일한 "없음" 메시지**로 처리한다. 남의 `accountId=999`를 찔러도 "권한 없음"이 아니라 "MarketAccount 없음"이 떠서, 어떤 ID가 실제로 존재하는지 추측할 수 없다. 모든 본인-리소스 API는 `accountId`를 직접 받지 않고 `/api/me/accounts` 목록에서 고르게 한다.

### 5.2 보안 — API 키는 응답/로그에 절대 노출 안 됨
`AccountView`는 `id/channel/vendorId`만 담는다(키 필드 자체가 없음). 동기화 실패 로그도 키가 아니라 메시지/transactionId만 남긴다. `MarketAccount`엔 `@ToString` 미사용.

### 5.3 정합성 — 멱등 수집
모든 수집은 UNIQUE 제약 + `existsBy...`로 멱등. 반품은 한 접수번호에 같은 상품 라인이 복수로 와도 **등장 순번(ordinal)** 을 외부키에 붙여 누락을 막는다(`ReturnExternalRefTest`로 단위 검증).

### 5.4 정합성 — 반품 이중 차감 방지
순이익 계산 시 **원가 기준 수량 = 주문수량 − 반품수량**(GREATEST 0)으로 깎되, 매출(payout)은 정산이 이미 반품을 음수로 반영하므로 추가 차감하지 않는다. 한쪽만 반영하면 이중 차감되는 함정을 명시적으로 피했다.

### 5.5 운영 — HTTP는 트랜잭션 밖
쿠팡 API 호출을 긴 단일 트랜잭션에 넣지 않는다. 수집(HTTP)은 트랜잭션 밖, 영속화는 `TransactionTemplate`로 페이지 단위. 외부 API 지연이 DB 커넥션을 잡아먹지 않게 했다.

### 5.6 운영 — 소스별 예외 격리
수동 동기화는 주문/정산/반품을 각각 try/catch로 격리해 `SyncResult{ok,count,error}`로 돌려준다. 한 소스가 실패해도 전체가 500으로 죽지 않고 **어디가 막혔는지** 화면/로그에 바로 보인다 — 라이브 키 검증에 결정적이었다(§6).

### 5.7 과금 — 플랜 한도 서버 강제
한도(FREE=1계정, PRO=무제한)는 `PlanType` 한 곳에 고정하고 **서버가 강제**한다. 화면은 한도 도달 시 폼을 잠그지만, 우회 호출도 서버가 막는다.

## 6. 실전 트러블슈팅 (실제 엔지니어링 스토리)

### 라이브 API 검증으로 잡아낸 정산 엔드포인트 경로 오류
수동 동기화 버튼을 만들고 **시드의 가짜 키(SEEDVENDOR)** 로 눌러봤더니, 쿠팡 실 API까지 실제로 도달했다. 소스별 격리 덕에 정확한 진단이 나왔다:

- **주문/반품 → 401 "Specified key is not registered"**
  → 경로·HMAC 서명 **구조는 정상**, 키만 가짜. 실 키면 동작 예상. ✅
- **정산 → 404 "No exactly matching API specification for `/api/v1/vendors/{vendorId}/revenue-history`"**
  → **정산 엔드포인트 경로 자체가 틀림.** 실 키와 무관하게 라이브 문서로 수정 필요. ⚠️

핵심: "전체 500"이었다면 그냥 "동기화 실패"로 끝났을 것을, **소스별 결과 격리** 설계 덕에 401(키 문제)과 404(경로 문제)를 구분해 **다음 작업 1순위**를 정확히 특정했다.

### Windows 인코딩 함정들
- `javac` 기본 charset이 CP949 → 한글 리터럴 깨짐. `build.gradle`에 `options.encoding='UTF-8'` 고정으로 해결.
- Windows `curl`로 한글 JSON POST 시 CP949 → UTF-8 에러. 코드 버그가 아니라 셸 인코딩 문제(브라우저 fetch는 정상)로 진단.

### Flyway 마이그레이션에 섞인 예시 쿼리
`V1__init.sql` 말미의 named-param 예시 쿼리가 실제로 실행돼 문법 에러 → 주석 안내로 교체.

## 7. 직접 실행 / 시연 (쿠팡 키 불필요, 명령 2개)

`seed` 프로파일은 포트(8088)·개발용 암호화 키가 박혀 있어 환경변수 없이 뜬다. 흑자 2 + 적자 1 샘플이 자동 시드된다.

```bash
docker compose up -d                                   # 로컬 Postgres (5433→5432)
./gradlew bootRun --args='--spring.profiles.active=seed'
```

→ `http://localhost:8088/` 접속 → 데모 계정 `demo@demo.local` / `demo1234` 로그인
→ **적자 상품 B가 빨간 배경 + `적자` 뱃지로 맨 위**에 보이면 정상.

> 프론트 개발(핫리로드): `cd frontend && npm run dev` → `http://localhost:5173` (/api 는 :8088 프록시)

## 8. 로드맵 (다음 단계)

1. **실 쿠팡 키 라이브 검증** — 연동 입구 완성됨. 실 vendorId/키로 `[검증 필요]` 마커(엔드포인트 경로·JSON 필드명·페이징 토큰) 확정. **정산 엔드포인트 경로 수정이 1순위**(§6).
2. **조회 기간 한도 게이팅** — 계정 수 한도는 적용 완료. 대시보드 from/to를 플랜 기간(FREE=30일)으로 제한.
3. **토스 빌링 마무리** — 실 키 주입 + 토스 SDK 카드등록 + 응답/에러 코드 라이브 확정.
4. **반품 사유 표준화** — 쿠팡 사유 코드 매핑, 기간 비교 추세.
