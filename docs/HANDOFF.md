# HANDOFF.md — 시니어 리뷰어 역할 인수인계

> 저장소 위치 권장: `docs/HANDOFF.md`, 그리고 **`CLAUDE.md` 맨 위에 아래 한 줄을 추가**:
> `> 작업 시작 전 docs/HANDOFF.md 를 반드시 읽는다. 이 문서는 "구현자"뿐 아니라 "비판적 리뷰어" 역할까지 스스로 겸하라는 지시다.`

## 배경 — 왜 이 문서가 필요한가

지금까지는 별도의 시니어 리뷰어(Claude, 채팅)가 작업지시서를 쓰고, Claude Code 가 구현하고,
다시 리뷰어가 브라우저로 결과를 검증하는 3단 구조였다. 이제부터 Claude Code 가 **구현자와
리뷰어 역할을 동시에** 맡는다. 이 문서는 리뷰어가 실제로 어떤 기준으로 검증해왔는지를
넘겨, 그 기준을 스스로에게도 적용하게 하기 위한 것이다.

**핵심 원칙: "끝났다"고 말하기 전에, 남의 작업을 검수하듯 자기 작업을 의심하라.**
지금까지 여러 번 있었던 패턴 — Claude Code 가 "AC 충족, 다 됐다"고 보고했지만 실제로는
일부 화면에 적용이 안 됐거나(T9 가 처음엔 대시보드에만 적용되고 광고ROI 에 빠짐), 지시
의도를 오독해 화면 카피를 잘못 만든 경우가 있었다. 이건 능력 문제가 아니라 **자기 검수
단계가 없었기 때문**이다. 아래 체크리스트를 항상 통과시킨다.

## 1. 작업 전 — 계획 단계

- 요청이 모호하면, 코드부터 짜지 말고 **먼저 계획을 텍스트로 요약**하고 진행한다
  (예: "이렇게 이해했고 이 순서로 하겠습니다" 한 문단).
- 기존 도메인 경계를 확인한다: 새 기능이 어느 패키지/컴포넌트에 속하는지, 기존 것을 고치는지
  새로 만드는지. `CLAUDE.md` 패키지 구조·`docs/DECISIONS.md` 를 먼저 훑는다.
- **숨은 상호작용을 찾는다.** 지금까지 실제로 있었던 함정들:
  - 광고비를 새로 추가할 때 기존 `Cost(AD)` 배분과 이중차감(D1)
  - 무상 플랜 지급이 빌링 스케줄러의 결제 시도와 충돌(D3)
  - 공용 컴포넌트 변경이 다른 화면(대시보드 vs 광고ROI)에 빠짐없이 적용됐는지
  새 기능을 만들 때마다 "이게 기존의 어떤 배치/스케줄러/공용 컴포넌트와 부딪히는가"를
  먼저 자문한다.

## 2. 작업 중 — 원칙

- **불변식은 테스트로 증명한다, 말로 하지 않는다.** "이중차감 없음" 같은 주장은 반드시
  단위테스트로 고정한다(`ProfitCalculationServiceAdInvariantTest`,
  `AdRoiMainDashboardConsistencyTest` 가 좋은 선례).
- **서버가 진짜 방어선, UI는 편의.** 권한·소유권·플랜 한도는 항상 서버에서 강제하고,
  프론트 숨김/리다이렉트는 UX 편의로만 취급한다(`AdminAccess`, `AccountAccess` 패턴 유지).
- **폼 필드·스코프는 최소로.** 요청받지 않은 필드·기능을 "이왕 하는 김에" 추가하지 않는다.
  범위 밖 아이디어는 코드 대신 `docs/BACKLOG.md` 에 적는다.
- **가짜 지표 금지.** 사용자·거래액·후기 등 실존하지 않는 사회적 증거를 어떤 화면에도
  넣지 않는다. 실사용자 0명인 동안은 얼리액세스 프레이밍만 쓴다.
- **결정과 이유를 `docs/DECISIONS.md` 에 기록.** "무엇을 했다"가 아니라 "왜 그렇게
  했다/다른 대안은 왜 버렸다"를 남긴다. 지금까지의 D1~D3 항목 수준의 밀도를 유지한다.

## 3. 작업 후 — 자기 검수 (가장 중요, 새로 생긴 습관)

완료 보고 전에 스스로에게 묻는다:

1. **AC 를 하나씩 실제로 재현했는가**, 아니면 "될 것 같다"고 판단만 했는가? 커맨드 실행
   결과·테스트 로그·화면 텍스트로 근거를 남긴다.
2. **이 변경이 닿아야 할 화면/엔드포인트를 전부 건드렸는가?** (예: 공용 컴포넌트라면
   그걸 쓰는 모든 페이지, API 라면 그걸 부르는 모든 클라이언트)
3. **회귀가 없는가?** 새 기능 때문에 기존 대시보드·기존 테스트가 깨지지 않았는지 직접
   확인한다(같은 값 재계산·기존 테스트 재실행).
4. **보안/권한 우회 경로가 남았는가?** 새 엔드포인트가 인증/role 체크를 빠뜨리지 않았는지.
5. **지시 문구를 오독할 여지는 없었는가?** 특히 지시서 안의 "내부 분류명"과 "실제 화면
   카피"를 혼동하는 사고가 있었다(회원가입 "안심 문구" 라벨 사건). 애매하면 구현 전에
   "이 텍스트가 화면에 그대로 노출되는 게 맞습니까?"처럼 되물어 확인한다 — 이건 무능이
   아니라 좋은 습관이다.

이 5개 중 하나라도 "확인 안 함"이면, 완료 보고 대신 그 부분부터 검증한다.

## 4. 새 기능/제품 아이디어 판단 프레임 (신규 요청이 들어올 때)

사용자가 새 아이디어를 던지면(예: 실시간 채팅, 배달 자영업 버전 등) 바로 만들지 말고
아래 순서로 판단하고, 그 판단을 사용자에게 먼저 보고한다:

1. **기존 자산 재사용 가능한가?** (도메인 지식·데이터 파이프라인·UI 패턴) 재사용 안 되면
   완전히 새 제품이라는 뜻 — 지금 리소스로 감당되는지 재고.
2. **데이터 접근성이 있는가?** 공식 API 가 있는지, 없으면 수기/CSV 로도 가치가 나오는지.
3. **지금 진행 중인 핵심 제품(seller-profit)의 유료 검증이 끝났는가?** 안 끝났으면,
   새 아이디어는 대부분 **지금 만들 게 아니라 백로그**로 보낸다. 예외는 "만들지 않아도
   되는 것"(예: 채널톡 임베드처럼 기성품으로 대체 가능한 것)뿐이다.
4. 만들지 않기로 하면, 근거와 함께 `docs/BACKLOG.md` 에 기록하고 사용자에게 왜
   미루는지 설명한다. "안 된다"가 아니라 "지금은 아니다, 이유는 X, 다시 볼 시점은 Y"로.

### 백로그 항목 예시 — 배달 자영업자 광고 ROI (2026-07-04 기록)

- **아이디어**: 배민/쿠팡이츠 사장님 대상, 광고비(우리가게클릭·맞춤형광고 등) 반영
  진짜 순이익 — seller-profit 과 같은 wedge.
- **검증된 것**: 페인포인트는 실재. 배민 "우리가게클릭" 광고로 매출 늘어도 실소득이
  줄어드는 사례가 언론에 보도됨(ROAS 함정과 동일 패턴).
- **보류 이유**:
  1. 데이터 접근성이 쿠팡보다 나쁨 — 배민/쿠팡이츠에 쿠팡 Open API 같은 공식 API 존재
     여부 미확인, 사장님 웹 화면(배민셀프서비스)만 확인됨. 자동 수집이 어려우면 이 제품의
     핵심 가치(자동화)가 줄어든다.
  2. 도메인 불일치 — 조민석 님은 이커머스 셀러 경험은 있으나 요식업 매장 운영 경험 없음.
     seller-profit 을 고른 근거였던 "불공정한 우위"가 여기선 없다.
  3. **가장 중요**: seller-profit 자체가 아직 유료 고객 0명 — 검증 전 두 번째 제품 착수는
     솔로 개발자의 유한한 집중력을 분산시킨다.
- **재검토 시점**: seller-profit 이 유료 전환(베타테스터 중 실 결제 최소 1건) 이후,
  "같은 패턴을 다른 자영업 버티컬로 확장할 가치가 있는가"를 재판단.

## 5. 지금까지 합의된 GTM 로드맵 (현재 상태 반영)

우선순위는 "기능 추가"가 아니라 "실사용자를 만드는 것"이다. 순서:

1. ~~T7 (메인 대시보드 광고비 반영)~~ 완료
2. ~~T8 (랜딩 페이지)~~ 완료
3. ~~T9 (기간 선택 UX)~~ 완료
4. ~~T10 (관리자/베타 지급)~~ 완료
5. ~~T11 (회원가입 보강)~~ 완료
6. ~~배포/호스팅~~ 완료(2026-07-04, D4~D6) — `https://sellerprofit.co.kr` 라이브
7. **다음: 베타 셀러 5~10명 모집 + 관리자 기능으로 PRO 지급 + 도그푸딩**(조민석 님 본인 쿠팡
   데이터로 사용) — 2026-07-07 D11 에서 "유료 모집보다 베타 우선" 확정, 토스 실 결제 연동은
   후순위로 재배치(`CLAUDE.md` "다음 단계 A")
8. 실시간 채팅은 **직접 구현 대신 채널톡(Channel.io) 임베드**로 대체(백로그 판단 프레임
   4번의 "만들지 않아도 되는 것" 사례)
9. 요금제 재검토 — 경쟁사(셀러팟) 대비 저평가 가능성, 유료 검증 후 상향 검토
10. 지불의사 검증(베타 셀러에게 "월 얼마면 낼래?" 질문) → 실 결제 전환

각 단계는 이전 단계가 실제로 끝났다는 근거(배포면 접속 가능한 URL, 검증이면 실사용자
반응)가 있어야 다음으로 넘어간다.

## 6. 소통 방식 (참고)

사용자는 그동안 채팅 Claude(전략/리뷰) ↔ Claude Code(구현) 사이를 오가며 지시서를
전달해왔다. 이 구조가 앞으로도 병행될 수 있으니, **`docs/DECISIONS.md` 를 계속 최신으로
유지**하는 것이 두 역할 간(또는 이 문서를 읽는 미래의 어떤 세션이든) 문맥을 잃지 않는
핵심 장치다. 결정 기록이 곧 기억이다.
