# 작업지시서 — 기간 선택 UX 개선 (Claude Code 용)

> 저장소 위치 권장: `docs/period-picker-tasks.md`
> 기존 규약 계승: React+Vite SPA, 같은 오리진 서빙(`static/`), 프론트 수정 후 `npm run build` 산출물 커밋.

## 배경 — 왜 고치나

지금 대시보드·광고 ROI의 기간 조회가 **"연도-월-일"을 손으로 입력**하는 방식이라 셀러에게 어색하고 느리다. 셀러가 실제로 원하는 건 특정 날짜가 아니라 **"이번 달", "지난 달" 같은 기간**이다. 프리셋 중심으로 바꾸고, 직접 선택이 필요할 때만 달력을 노출한다.

## 목표 UX

기본 화면에 **기간 프리셋 칩**을 노출한다:

```
[오늘] [이번 주] [이번 달] [지난 달] [최근 7일] [최근 30일] [직접 선택]
```

- 칩 선택 → `from`/`to`를 자동 계산·세팅하고 **바로 조회**(별도 "조회" 클릭 불필요).
- 기본 선택 = **"이번 달"**(또는 플랜 기본 조회기간에 맞춤).
- **"직접 선택"을 누른 경우에만** 날짜 범위 선택 UI(달력) 노출. 텍스트 수기 입력 제거.

## T9 — 공용 PeriodPicker 구현

### 9.1 공용 컴포넌트
- `frontend/src/components/PeriodPicker.jsx` 신설. props: `value({from,to,preset})`, `onChange`, `maxRangeDays`(플랜 한도), `disabled`.
- 프리셋 계산은 **KST(Asia/Seoul) 기준**으로. (기존 쿠팡 데이터가 KST 기준이므로 일관성 유지.)
  - 오늘 = [오늘 00:00, 오늘]; 이번 주 = [이번 주 월요일, 오늘]; 이번 달 = [1일, 오늘]; 지난 달 = [지난달 1일, 지난달 말일]; 최근 7/30일 = [오늘−N+1, 오늘].
- 선택된 프리셋 칩은 활성 상태로 표시(현재 선택이 무엇인지 보이게).

### 9.2 직접 선택(달력)
- "직접 선택" 시에만 **날짜 범위 달력** 노출. 텍스트 입력이 아니라 **시작~종료를 클릭으로** 고르는 range picker.
- 가벼운 라이브러리 사용 가능(`react-day-picker` 권장). 한국어 로케일·월 이동 지원.
- 시작일 > 종료일 등 잘못된 범위는 UI에서 차단(기존 서버 CHECK 규칙과 동일 방향).

### 9.3 플랜 한도 연동 (겸사겸사 처리)
- `PlanType.dashboardLookbackDays`(예: FREE=30일)를 `maxRangeDays`로 전달.
- 한도를 넘는 프리셋(예: FREE에서 "최근 30일" 초과분)은 **비활성/숨김** 처리하거나, 직접 선택에서 한도 초과 범위 선택 시 안내 문구 노출("현재 플랜은 최근 30일까지 조회할 수 있어요 · PRO로 업그레이드").
- ⚠️ 이건 CLAUDE.md "다음 단계 A"의 **미적용 항목(조회 기간 한도 게이팅)** 과 맞물린다. **UI 차단만으로 끝내지 말고 서버 게이팅과 일치**시킬 것(대시보드 from/to 가 플랜 기간 초과 시 서버가 자르거나 거부). 서버 강제가 진짜 게이트, UI는 안내.

### 9.4 적용
- **대시보드**(`/dashboard`)와 **광고 ROI**(`/ad-roi`) 양쪽의 기존 "시작일/종료일" 입력을 PeriodPicker로 교체. 두 화면이 같은 컴포넌트 재사용.
- 조회 API 호출(`/api/dashboard/profit`, `/api/dashboard/ad-roi`)의 from/to 파라미터 형식은 기존 그대로 유지(백엔드 변경 최소화).

### 9.5 품질 기준
- 반응형: 모바일 폭에서 칩이 줄바꿈되며 눌리기 쉬운 크기 유지.
- 접근성: 칩은 실제 버튼(키보드 포커스·엔터 동작), 달력은 키보드 탐색 가능, `prefers-reduced-motion` 존중.
- 프리셋 라벨은 사용자 언어로(시스템 용어 금지).
