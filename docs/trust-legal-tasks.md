> 저장소 위치 권장: `docs/trust-legal-tasks.md`
> 기존 규약 계승: 프론트 수정 후 `npm run build` 산출물 커밋, 페이퍼/잉크 디자인 톤 유지,
> 가짜 지표 금지 원칙(`docs/HANDOFF.md`), `docs/onboarding-tasks.md`(T13)와 병행 가능.

## T14 — 개인정보처리방침 · 이용약관 (법적 필수, 최우선)

### ⚠️ 중요한 전제

**Claude Code는 변호사가 아니고, 나도 아니다.** 아래는 통상적인 국내 서비스의 표준 구성을
안내하는 것이지 완성된 법률 문서가 아니다. 실제 게시 전에는 **개인정보보호위원회
(privacy.go.kr)가 배포하는 표준 개인정보처리방침 템플릿**을 내려받아 그 틀에 맞춰
작성하는 것을 권장한다(Claude Code가 임의로 법 조문을 창작하지 않는다). 초안 작성 후
**실제 게시 전 조민석 님이 한 번 더 확인**하고, 필요하면 전문가(법무사·변호사) 검토를
받는다.

### T14.1 — 페이지 신설

- `/privacy`(개인정보처리방침), `/terms`(이용약관) 라우트 신설.
- 개인정보처리방침에 최소 포함할 항목(표준 템플릿 기준 구조만 잡고, 조민석 님이 채울 것은
  `[작성 필요]`로 표시):
  1. 수집하는 개인정보 항목 (이메일, 비밀번호(해시), 쿠팡 API 키(암호화 저장) 등)
  2. 수집 목적 (회원 관리, 서비스 제공, 쿠팡 데이터 연동)
  3. 보유·이용 기간 (회원 탈퇴 시 파기 등 원칙)
  4. 제3자 제공 여부 (원칙적으로 없음, 있다면 명시)
  5. 처리 위탁 (호스팅사 iwinv 등 인프라 위탁 사실 명시)
  6. 정보주체(이용자)의 권리와 행사 방법 (열람·정정·삭제·처리정지 요구권)
  7. 개인정보 파기 절차
  8. 개인정보 보호책임자 연락처 `[작성 필요: 조민석 님 연락처/이메일]`
  9. 시행일자
- 이용약관: 서비스 이용 조건, 회원 의무, 면책 조항 등 표준 구성 `[작성 필요]` 로 골격만.
- 두 페이지 모두 **랜딩 푸터**와 **회원가입 화면**에서 링크로 접근 가능해야 한다.

### T14.2 — 회원가입 동의 체크박스

- `Signup.jsx` 에 **"[필수] 개인정보 수집·이용에 동의합니다"** 체크박스 추가(`/privacy`
  링크 포함), **체크 안 하면 가입 버튼 비활성화**.
- (선택) 이용약관 동의도 별도 체크박스로, 또는 하나로 묶어도 됨 — 국내 서비스 관행상
  분리가 일반적이나 초기 베타는 하나로 묶어도 무방. 조민석 님 결정 필요 시 `docs/DECISIONS.md`
  에 근거와 함께 기록.
- **폼 필드 자체(이메일/비밀번호)는 그대로 유지** — T11 규칙(필드 늘리지 않기) 계승,
  체크박스는 예외로 허용(법적 필수 요소이므로).

### AC (T14)

- [ ] `/privacy`, `/terms` 접속 가능, 랜딩 푸터·회원가입 화면에서 링크로 도달 가능.
- [ ] 회원가입 시 동의 체크 안 하면 제출 불가.
- [ ] `[작성 필요]` 표시된 항목 목록을 완료 보고에 정리해서 조민석 님이 직접 채워야 할
      부분을 명확히 전달.

## T15 — 신뢰·지원 빠른 보강 (저비용, 베타 전 권장)

### T15.1 — 문의 채널 (최소 버전)

- 대시보드/계정연동/광고ROI 등 주요 화면 하단 또는 nav 에 **"문의하기"** 링크 추가.
  베타 단계에선 채널톡 같은 도구 없이 **`mailto:` 링크**로 충분
  (예: `mailto:{조민석 님 이메일}?subject=seller-profit 문의`).
- 계정 연동 화면(T13.1 IP 안내 근처)에 "연동이 안 되면 문의하기" 형태로 배치하면
  가장 효과적(문제 생기는 지점 바로 옆).

### T15.2 — 비밀번호 재설정 흐름 확인

- 현재 비밀번호 재설정 기능이 있는지 먼저 확인(코드베이스 검색). **없으면 최소 버전**으로:
  이메일 입력 → 재설정 링크 발송 → 새 비밀번호 설정. (이메일 발송 인프라가 없으면
  범위를 줄여, 최소한 "비밀번호를 잊으셨다면 문의하기"로 T15.1 문의 채널을 임시
  안내로 사용 — 베타 규모에서는 이 정도로 충분, 정식 기능은 후속 백로그로.)
- **AC:** 재설정 기능 유무를 확인한 결과를 보고. 있으면 정상 동작 재현, 없으면 임시
  안내(문의 채널 연결)라도 반영.

### T15.3 — 사소한 마감

- 랜딩을 포함한 모든 페이지의 `<title>` 이 페이지 성격에 맞게 설정(현재 랜딩도
  "…대시보드"로 고정된 문제 확인 필요) — 예: 랜딩은 "seller-profit — 쿠팡 셀러 진짜
  순이익", 대시보드는 기존 유지.
- 파비콘 존재 여부 확인, 없으면 간단한 파비콘 추가(브랜드 이니셜 등 최소 수준으로).

### AC (T15)

- [ ] 문의 링크가 최소 계정연동 화면에 존재하고 mailto 정상 동작.
- [ ] 비밀번호 재설정 현황 확인 및 최소 조치 반영.
- [ ] 페이지별 `<title>` 정상, 파비콘 존재.

## T16 — 샘플 데이터 체험 모드 (전환율 개선, 선택이지만 권장)

### 배경

지금은 실제 쿠팡 Secret Key를 넣어야만 뭔가를 볼 수 있다. 민감한 키를 넣기 전에 "이게
진짜 도움이 되는지" 확신이 없는 방문자가 이탈할 수 있다. 로그인·API 키 없이 실제
대시보드 모습을 미리 보여주는 체험 모드로 심리적 장벽을 낮춘다.

### ⚠️ 규칙

- 이건 **명확히 "샘플 데이터"로 라벨링**한다 — 실사용자 통계인 것처럼 보이면 안 된다
  (가짜 지표 금지 원칙과 다른 문제: 이건 라벨이 명확하면 허용되는 데모 데이터).
- **기존에 이미 쓰고 있는 예시(적자상품 B/흑자상품 A·C, 랜딩·ReceiptCard 의 숫자)를
  재사용**한다 — 새로운 가짜 수치를 창작하지 않는다. 일관성 유지.
- 실제 백엔드/DB 에 전혀 접근하지 않는 **순수 프론트엔드 정적 렌더링**으로 구현
  (로그인 불필요, 상태 없음).

### 구현

- 랜딩의 "무료로 내 순이익 보기" 버튼 근처에 **"샘플로 먼저 보기"** 링크/버튼 추가.
- `/demo`(또는 `/sample`) 라우트: 로그인 없이 접근 가능. 대시보드·광고ROI 와 유사한
  레이아웃에 기존 예시 숫자(적자상품 B −159,737 등)를 정적으로 채워서 렌더.
- 화면 상단에 눈에 띄게 **"샘플 데이터입니다 · 실제 내 데이터를 보려면 무료 가입"**
  배너 + CTA 버튼(`/signup`).

### AC (T16)

- [ ] `/demo` 비로그인 접근 가능, 기존 예시 숫자와 일관됨(새 수치 창작 없음).
- [ ] "샘플 데이터" 라벨이 화면 전체에 명확히 보임(혼동 불가능한 수준).
- [ ] 실제 API 호출 없음(순수 정적 렌더링), 백엔드 부하 없음.
- [ ] 가입 유도 CTA 정상 동작.
